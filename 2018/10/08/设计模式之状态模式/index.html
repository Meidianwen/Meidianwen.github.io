设计模式之状态模式

发表于 2018-10-08 | 分类于 设计模式 | | 阅读次数

状态模式

在状态模式中，类行为根据其状态而改变。 这种设计模式属于行为模式。在状态模式中，我们创建表示各种状态的对象，以及随状态对象变化而行为变化的上下文对象。

什么时候使用此模式

  1. 当对象的行为依赖于状态，而该对象必须根据其状态（在运行时)改变其行为
  2. 当操作带有大量状态相关的、多部分的条件语句

设计


可以创建一个Context上下文类，此类用来具体和状态类交互，这样可以实现客户程序不需要直接与状态对象交互。
把状态做一个抽象类，然后具体的状态子类继承自抽象类，在子类中封装了不同状态下的行为
context与状态类是协作关系：

  1. Context代表状态相关的请求。
  2. Context可将自己作为一个对象通过参数传递给状态对象，从而可让状态对象访问Context类的方法
  3. 客户对象先创建一个具体的状态子类的对象s，然后，在创建Context对象时，通过参数将s传递给Context对象。此后，客户程序就不必与该状态对象直接交互。
  4. 状态转换： Context类或具体的状态子类负责

状态转换的建议：

  1. 客户类只创建一个“种子对象”，然后，传递给Context对象；
  2. 再由Context对象根据状态变化的情况，负责创建所有的其它的对象

状态模式的优点

优点：

  1. 容易增加新的状态
  2. 是状态转换更加明确

状态模式与策略模式的区别


状态模式与策略模式的区别：

  1. 状态模式中子类都是一些状态相关的行为，而策略模式都是一些不同的算法
  2. 状态模式中状态对象可以负责更新配置给Context类的状态对象；而策略模式是由客户端创建策略子类且传递给Context
  3. 状态的选择依赖于Context或State对；策略模式由应用类 选择策略子类
  4. 状态模式有状态转换；而策略模式没有
  5. 两者都可以将一些逻辑代码写道Context中，但是状态模式中Context中科以有状态转换，而策略模式没有，策略不是不会自己转换，只能通过应用类来选择。
